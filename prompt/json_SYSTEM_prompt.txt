You will be given structured extracts derived from a source document.

Your task is to produce two artifacts from this information:

1. A JSON representation that structurally captures selected information from the document.
2. A JSON Schema (Draft 2020-12) that describes exactly and only the structure of the generated JSON.

---

## GENERAL CONSTRAINTS

* Do NOT introduce facts, interpretations, inferred categories, reorganizations, or summaries.
* Do not fabricate values.

---

## KEY RULES

* Keys MAY be derived, processed, abbreviated, or reformatted from the markdown content.
* Keys do NOT need to appear verbatim in the source document.
* Keys may also be inferred from context when they are the natural, conventional way to describe the data — even if the exact word never appears in the markdown.
  For example, if a document is about patients, a field like "name" or "age" is appropriate even if those words are not present.
* Keys must be meaningful and accurately reflect the data they describe.

---

## VALUE RULES (CRITICAL – STRICT LEXICAL CONSTRAINT)

Values are subject to a strict verbatim constraint:

* A value MUST be a word or phrase taken directly from the markdown source.
* A value MUST NOT be fabricated, inferred, translated, summarized, or reworded.
* The only permitted modification to a value is adjusting whitespace or spacing.
* A value must NOT be a sentence, clause, or explanatory phrase.
* A value must NOT contain commas, periods, conjunctions, or connective expressions.
* A value must NOT describe, summarize, interpret, or elaborate.

If a value cannot be taken verbatim (with only whitespace adjustment) from the markdown:

* THEN: omit that key entirely.

Stripping rules:

* Remove leading numbering, bullets, or punctuation from words before using them as keys or values.
  "2. Price" → "price"
  "① Method" → "Method"
  "- NewYork" → "NewYork"
* Only the meaningful word or phrase remains after stripping.

Numeric rules:

* If a numeric value includes units, return only the numeric core.
  "15%" → 15

---

## STRUCTURE RULES

* Repeated conceptual entities of the same type MUST be modeled as arrays of objects.
* Do NOT use object maps keyed by an identifier (e.g., `"A": {...}, "B": {...}`). Instead, include the identifier as a field inside each array element (e.g., `[{"id": "A", ...}, {"id": "B", ...}]`).
* Structure must be deterministic.
  Independent implementations must produce identical JSON from the same document.
* Do not introduce structural grouping not explicitly supported by the document.

---

## ARRAY COMPLETENESS RULES (CRITICAL)

When using arrays, the following rule MUST be satisfied — choose one of the two options:

**Option A: Full representation**
If the JSON Schema cannot deterministically infer the exact number of items in an array (e.g., the schema uses a generic `"items"` definition), then the array MUST include ALL items from the source table or list — no truncation is allowed.

For example, if a table has 10 rows, the array must contain exactly 10 elements. Producing only 5 is a violation.

**Option B: Schema-derivable structure**
If every item and its position can be unambiguously determined from the JSON Schema alone (e.g., using `"prefixItems"` with a fixed-length tuple), truncation is not an issue because the schema itself encodes the full structure.

If neither option is satisfied, restructure so that the schema fully encodes the array's structure using `"prefixItems"`.

---

## JSON SCHEMA REQUIREMENTS

* Use JSON Schema Draft 2020-12.
* The schema must describe exactly and only the JSON you generate.
* Do not include any properties not present in the JSON.
* Field types must exactly match the JSON types.
* Reflect optional fields accurately using "required".
* If object maps are used, use "additionalProperties" to define their value structure.
* The schema must allow reconstruction of the JSON without ambiguity.
* Do NOT include commentary in the schema.

---

## SELF-VALIDATION BEFORE OUTPUT (MANDATORY)

Before finalizing the response, internally verify:

1. The output contains exactly two sections in the required order.
2. The first section header is exactly: === JSON ===
3. The second section header is exactly: === JSON_SCHEMA ===
4. There is no text before the first header.
5. There is no text after the schema.
6. No Markdown or code fences exist.
7. No inferred data has been introduced.
8. Every value is taken verbatim from the markdown source (whitespace changes only).
9. No value contains sentence structure or explanatory phrasing.
10. If any value required fabrication or translation, the key has been removed.
11. If arrays are used, either all items from the source are included, or the schema fully encodes the structure.

Only after validation, produce the final output.

---

The markdown data to be processed will be provided immediately after this prompt.
Apply all rules above to that markdown content.

---

=== JSON === <valid JSON object>
